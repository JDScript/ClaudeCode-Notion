# Portable Notion Skills System — Design

Date: 2026-02-25

## Problem

Skills are not portable across Notion accounts. 30 hardcoded UUIDs across 11
files tie the system to a single workspace. Per-project template customization
is not supported. No onboarding flow exists for new users.

## Solution

Three changes: a local gitignored config file, an `init` skill that discovers
the workspace, and a per-project template override mechanism stored in Notion.

---

## 1. Config File

**Path:** `.claude/notion-config.json` (gitignored)

Generated by `init`, read by all skills at runtime.

```json
{
  "workspace_root": "<page-id>",
  "projects": {
    "Project Name": {
      "page_id": "<page-id>",
      "tasks_db": "<collection-id>",
      "references_db": "<collection-id>",
      "datasets_db": "<collection-id>",
      "meeting_notes": "<page-id>",
      "templates": "<page-id>"
    }
  }
}
```

Only discovered fields are included — no nulls for missing DBs.

## 2. `init` Skill

**Trigger:** `/init` or first-time setup.

**Input:** A Notion page URL (the workspace root).

**Steps:**

1. Fetch the workspace root page via `notion-fetch`
2. Discover all child pages (projects) from the page content
3. For each project, fetch and identify inline DBs (Tasks, References, Datasets)
   and sub-pages (Meeting Notes, Templates)
4. If `.claude/notion-config.json` exists, merge — add new projects, update
   changed IDs, preserve manually-added entries
5. Write config to `.claude/notion-config.json`
6. Update `CLAUDE.md` with the workspace root ID
7. Report what was discovered

**Re-run behavior:** Merges new discoveries into existing config. Never deletes
entries (user may have manually added config for pages `init` can't
auto-discover).

## 3. Hardcoded ID Removal

### Skills

`scaffold-project`, `manage-tasks`, `read-dataset`:
- Replace all hardcoded IDs with instructions to read from config
- Example IDs become clearly marked `<placeholder>` examples
- Skills fail gracefully with "Run `/init` first" if config missing

### Docs

- `docs/samples/*.md`: replace specific collection IDs with `<collection-id>`
- `docs/architecture/overview.md`, `docs/skills/catalog.md`: replace page IDs
  with references to config
- `docs/plans/*`: leave as-is (historical documents)

### CLAUDE.md

- Keep workspace root ID (updated by `init`)
- Remove all other project-specific IDs

## 4. Per-Project Templates

**Storage:** A "Templates" sub-page under a project in Notion, containing child
pages named by template type (e.g., "Paper Summary Template").

**Fallback chain:**

1. Check config for project's `templates` page ID
2. If present, fetch from Notion → use project-specific template
3. If absent, use `docs/templates/` local defaults

**Creation:** Only when a user explicitly creates one. `scaffold-project` does
NOT auto-create template pages.

**Discovery:** `init` detects existing Templates sub-pages and records their
page ID in config.

## 5. Skill Config Reading Pattern

Every skill that needs IDs adds this as its first step:

> **Step 1: Load config**
> Read `.claude/notion-config.json`. If file not found, stop and tell the user
> to run `/init` first.

This replaces the current hardcoded ID pattern.
